---

## 1. Суть проекта

**ClipQueue** — минималистичный Windows desktop инструмент, который превращает многострочный текст в очередь буфера обмена. Пользователь нажимает Ctrl+V несколько раз подряд — каждый раз в буфер подставляется следующий элемент очереди. Никакого alt+tab, никаких ручных копирований.

**Аналогия:** Clipboard history manager наоборот — не хранит прошлое, а распределяет будущее.

---

## 2. Проблема

При data entry задачах (перенос данных из чата/почты в Excel/CRM/формы) пользователь вынужден:
1. Alt+Tab в источник
2. Найти нужный элемент мышкой
3. Выделить → Ctrl+C
4. Alt+Tab обратно
5. Ctrl+V

При 20+ элементах это 20× прерываний контекста. ClipQueue убирает шаги 1-4 полностью — остаётся только Ctrl+V.

---

## 3. Пользователь

**Первичный:** Люди с монотонными data entry задачами — менеджеры, HR, операторы CRM, QA инженеры тестирующие формы.

**Вторичный:** Разработчики, которым нужно вставлять разные тестовые значения по очереди.

**Техническая грамотность:** Средняя. Должны уметь запустить .exe. Не должны знать Python.

---

## 4. Конкуренты

| Инструмент | Что делает | Почему не то |
|------------|-----------|--------------|
| Ditto / CopyQ / ClipClip | Clipboard history | Хранит прошлое, не управляет очередью |
| AutoHotkey | Может сделать то же самое | Требует скриптинга, барьер входа высокий |
| PureText | Вставка без форматирования | Другая задача |

**Вывод:** Ниша пустая для non-technical пользователя.

## 6. Основной алгоритм (ядро)

```
ИНИЦИАЛИЗАЦИЯ:
  items = []
  idx = 0
  active = False

ЗАГРУЗКА (кнопка Load или хоткей):
  raw_text → parser(strategy) → items[]
  idx = 0
  active = True
  clipboard ← items[0]  # предзагрузить первый элемент

ПЕРЕХВАТ Ctrl+V:
  IF NOT active OR items empty:
    pass_through()  # обычный Ctrl+V
    return
  
  # Элемент уже в буфере (предзагружен)
  pass_through()    # отправить Ctrl+V приложению
  sleep(15ms)       # ждать пока приложение прочитает буфер
  
  idx += 1
  IF idx < len(items):
    clipboard ← items[idx]  # предзагрузить следующий
  ELSE:
    active = False
    notify_user("Очередь завершена")

КРИТИЧНО: предзагрузка (не подмена в момент нажатия) решает race condition.
```

**Ключевой инсайт:** Буфер обновляется ПОСЛЕ вставки, а не до. Это надёжнее чем suppress → подмена → release.

## 7. Парсер (стратегии)

```python
strategies = {
    "lines":     lambda t: [l.strip() for l in t.splitlines() if l.strip()],
    "sentences": lambda t: re.split(r'(?<=[.!?])\s+', t.strip()),
    "comma":     lambda t: [x.strip() for x in t.split(',') if x.strip()],
```

## 8. UI Спецификация

### Размер и поведение
- Окно: `280×180px` в свёрнутом состоянии, `280×320px` в режиме ввода
- `always on top: True`
- `no_activate: True` — окно НЕ должно перехватывать фокус при нажатии хоткея
- Прозрачность: `85%` когда не в фокусе, `100%` когда в фокусе
- Позиция по умолчанию: правый нижний угол экрана

### Состояние 1 — Input Mode

## 9. Технические риски и решения

| Риск | Вероятность | Решение |
|------|-------------|---------|
| Race condition (приложение читает старый буфер) | Средняя | Предзагрузка + `sleep(15ms)`, настраиваемый delay |
| Focus steal (вставка идёт в само окно ClipQueue) | Высокая | `wm_attributes('-topmost')` + `focus_force` отключить на хоткей момент |
| Конфликт с PowerToys / AHK | Низкая | Документировать, дать возможность сменить хоткей |
| Admin права (keyboard либа) | Высокая | В README явно указать, добавить UAC manifest в PyInstaller |
| Excel/SAP нестандартная вставка | Низкая | Настраиваемый sleep до 50ms |
