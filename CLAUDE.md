# ClipQueue — Claude Code Instructions

## Правило №0 — читай первым
Прочитай `PRD.md` полностью перед любым действием.
`PRD.md` — единственный источник истины. Если чего-то нет в PRD — спроси, не придумывай.

---

## Режим работы: Plan → Confirm → Execute

**Никогда не пиши код сразу.**

Для каждой задачи обязательный порядок:

### Шаг 1 — Plan
Прежде чем писать код, опиши план:
- Что именно будет сделано
- Какие файлы будут созданы или изменены
- Какие зависимости затрагиваются
- Какие риски есть (из PRD раздел 9)

Формат:
```
📋 ПЛАН: [название задачи]

Файлы:
  + создать: core/queue.py
  ~ изменить: main.py (добавить импорт)

Что будет сделано:
  1. ...
  2. ...

Риски: [если есть из PRD]

Подтверди: продолжать? (да / скорректировать)
```

### Шаг 2 — Confirm
Жди подтверждения. Не начинай реализацию пока не получишь "да" или скорректированные требования.

### Шаг 3 — Execute
Реализуй строго то что описал в плане. Не добавляй ничего лишнего.

## Чеклисты

После завершения каждой фазы сгенерируй чеклист проверки на основе функционала фазы.
Формат:

```
## Чеклист Фазы N — [название]

### Функциональность
- [ ] [конкретное поведение которое можно проверить руками]
- [ ] ...

### Приложения
- [ ] Работает в Notepad
- [ ] Работает в Excel
- [ ] Работает в Chrome (поле ввода)

### Качество кода
- [ ] @code-quality-guardian прогнан
- [ ] Нет очевидных edge cases без обработки
- [ ] Коммит сделан
```

Каждый пункт проверяется вручную или агентом перед переходом к следующей фазе.

## Суб-агенты

Установи из `github.com/VoltAgent/awesome-claude-code-subagents`.

| Задача | Агент |
|--------|-------|
| Структура проекта, requirements.txt | `@project-setup-wizard` |
| core/queue.py, core/parser.py | `@python-expert` |
| ui/window.py, layout, цвета | `@ui-designer` |
| WinAPI, UAC, хоткей конфликты | `@windows-automation-specialist` |
| Race condition, sleep тюнинг | `@performance-profiler` |
| Перед каждым коммитом | `@code-quality-guardian` |

Вызывай агента в начале Plan шага для соответствующей задачи.

## Стиль кода

- Короткие имена: `idx`, `res`, `q`, `buf`
- Комментарии только про *почему*, никогда про *что*
- Без type hints везде подряд
- Без `# === SECTION ===` разделителей
- Не обрабатывай edge cases которые не нужны прямо сейчас
- Меньше кода > больше кода
